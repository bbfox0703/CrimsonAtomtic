# UI / UX design

## Goal

A save editor that loads fast, makes the common edit path obvious, and never makes the user wonder where to start.

## Anti-patterns from the old reference repo

Calling these out explicitly because the new UI must avoid them by design, not by accident.

1. **Tab soup on launch.** The old app shows 16 tabs immediately, some half-deprecated (DMM WebView, legacy compat). New users don't know where to start.
2. **Single 4,173-line MainWindow.** Everything is bound into one class; no clear separation of view, view-model, and data.
3. **Cluttered startup screen.** The first thing the user sees is config UI + game-path picker + tab strip, with no obvious "open my save" path.
4. **Slow load.** 45 JSON files in `data/`, several MB each, are decoded on first tab open. The biggest is a 21.8 MB world map JSON. Most of it isn't needed before the user has selected a save.
5. **Full-size icons rendered at 32 px.** `icon_cache.py` loads WebP at original resolution into QPixmap. Memory and bandwidth wasted across thousands of items.
6. **Synchronous network fetches for missing icons.** Threaded, but still slow and visible to the user.
7. **In-place dict mutation as edit model.** Hard to undo, hard to validate, hard to type-check.
8. **Duplicate / shadow data files** (e.g. `items.jsonl` vs `item_templates.json`) cause UI labels and tooltips to disagree depending on which file a code path read.

## Principles for the new UI

### Startup must be fast

- **First paint < 500 ms** on a modern dev machine.
- No game-data parsing on startup. The app comes up with an empty workspace and a single, obvious **Open save** action.
- Game-data tables (iteminfo, paloc, skill) load **lazily on first need**, in the background, with a visible progress indicator.

### Single dominant action per screen

- Launch → empty workspace + Open save (or recent-saves list, if a recent file exists).
- After open → the save shows up as a left-rail navigator (Inventory / Skills / Stats / World / …), not a tab strip.
- Each pane has one primary action and a small set of secondary ones, not a wall of buttons.

### Lazy and incremental

- Inventory list virtualised — no eager rendering of 6,000+ items.
- Icons rendered from a thumbnail cache, not the source WebP. Cache lives under `%LOCALAPPDATA%\CrimsonAtomtic\IconCache\` and is regenerated from game extraction, not from the old repo.
- Localized strings resolved on demand from PALOC; the language switch is a UI control, not an app restart.

### Typed model, undo-friendly

- The C# `SaveModel` is the only thing the UI talks to. Each edit is a typed command with `Apply` + `Undo` methods.
- The C# layer never holds raw bytes; raw bytes live behind the FFI in `crimson-rs`.

### Localisation

- **UI shell strings**: English only, in `Resources/Strings/en.axaml`, via `StaticResource` bindings. (CLAUDE.md rule.)
- **In-game item / skill / region names**: resolved from PALOC at display time, in whichever language the user picked. The shell language and the in-game language are independent settings.

## Asset pipeline

```
old repo icons_local/*.webp  ─┐
                              ├─► thumbnail pipeline ─► IconCache/  (cached on disk,
game-extracted icons         ─┘  (Pillow / SkiaSharp,                ETag = source SHA)
                                 32×32 + 64×64 + 128×128,
                                 PNG or AVIF)
```

- **One-time mining** from the old repo (icons only, no scripts).
- After the mining is done, the only icon source going forward is in-game extraction.
- Thumbnails are generated by an offline tool (`tools/extract/build_iconcache.py`) and shipped with the AOT binary as a starter cache. The user can rebuild on demand for a newer game version.

## Open questions

- Theme: light / dark / system. Default to system. Implementation: Avalonia FluentTheme.
- Save backups: auto-backup on every load? On every write? Both, separate retention policy?
- Mod awareness: how to surface "this item came from a mod" without cluttering vanilla item display.
- Multi-slot view: should the user see all 100+ slots at once, or pick one slot at a time? Probably a slot picker on launch.

These get resolved as we build, but write the answer here once decided so the rationale doesn't evaporate.
